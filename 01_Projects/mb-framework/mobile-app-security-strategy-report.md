# 모바일 앱 보안 전략 보고

경영진 보고용 **비기술·정책/리스크 중심** 요약. (React Native 기반 Android / iOS 통합 대응) — **MB Framework** 보안 전략.

---

## 1️⃣ 추진 배경

당사는 **React Native 기반으로 Android 및 iOS를 동시에 지원**하는 모바일 금융 서비스(**MB Framework**)를 구축 예정입니다.

인도네시아 금융감독청 **Otoritas Jasa Keuangan (OJK)** 기준 충족 사항:

- 루팅(Android) / 탈옥(iOS) 단말 차단
- 무결성 위반 단말에서 금융 서비스 제한
- 위변조 및 보안 우회 위험 최소화
- 민감 기능 사용 시 강화된 보안 검증

---

## 2️⃣ 왜 루팅/탈옥 감지가 중요한가

루팅/탈옥 단말 위험:

- 앱 위변조 가능
- 계정 탈취 및 세션 하이재킹 위험 증가
- 키로깅/메모리 후킹 가능
- OTP 및 인증 우회 가능
- 금융 거래 조작 가능성

👉 **금융 앱의 기본 전제는 “신뢰 가능한 단말”입니다.**

---

## 3️⃣ 기술 전략 개요 — 2단계 보안 구조

### 🔹 1단계: 즉시 탐지 (클라이언트)

React Native 라이브러리 기반 감지:

- 루팅 / 탈옥 여부
- USB 디버깅 활성화 여부
- 디버그 모드 실행 여부

특징: 앱 실행 즉시 차단 가능, 정책 기반 경고/차단, 구현 난이도 낮음, 빠른 적용 가능.

### 🔹 2단계: 공식 무결성 검증 (서버 검증형)

| 플랫폼 | API | 특징 |
|--------|-----|------|
| **Android** | Google Play Integrity API | Google 서버 기반 검증, 기기 무결성 등급, 우회 난이도 높음, 금융권 수준 적합 |
| **iOS** | Apple App Attest | Apple 서버 기반 무결성 검증, 앱 위변조 방지, 기기 신뢰도 검증 |

---

## 4️⃣ 감지 방식 비교

| 구분 | 1단계 (로컬 탐지) | 2단계 (공식 API 기반) |
|------|-------------------|------------------------|
| 루팅/탈옥 감지 | 가능 | 간접 판정 |
| USB 디버깅 감지 | 가능 | 불가 |
| 디버그 모드 감지 | 가능 | 불가 |
| 기기 무결성 등급 | 불가 | 가능 |
| 서버 검증 | 없음 | 있음 |
| 우회 난이도 | 중간 | 매우 높음 |

---

## 5️⃣ 최종 권장 전략 (경영 판단용)

### ✅ 혼합 전략

1. **앱 실행 시** → 루팅/탈옥 즉시 차단  
2. **로그인/이체/카드관리 등 민감 기능 사용 시** → 서버 기반 무결성 검증 통과 필수

---

## 6️⃣ OJK 대응 관점

본 전략으로 충족하는 항목:

- ✔ 루팅/탈옥 단말 탐지 및 차단
- ✔ 단말 무결성 검증 체계 존재
- ✔ 서버 기반 보안 검증 로직 운영
- ✔ 금융거래 시 강화 인증 정책 적용 가능

👉 “탐지 기능 존재”가 아니라 **“정책 기반 통합 보안 아키텍처”** 운영.

---

## 7️⃣ 기대 효과

- 금융 사고 리스크 감소
- 계정 탈취 방지
- 위변조 앱 차단
- 감독기관 대응 논리 확보
- 향후 보안 감사 대응 용이

---

## 8️⃣ 구현 복잡도 및 일정 영향

| 구분 | 영향도 |
|------|--------|
| 1단계 적용 | 낮음 (빠른 적용 가능) |
| 2단계 적용 | 중간 (서버 검증 로직 필요) |
| 전체 일정 영향 | 제한적 (설계 초기 반영 시 문제 없음) |
